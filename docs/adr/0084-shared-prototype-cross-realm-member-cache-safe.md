# Cross-realm prototype member cache is safe for TGocciaSharedPrototype units

**Date:** 2026-06-29
**Area:** `engine`
**Issue:** follow-up to [#892](https://github.com/frostney/GocciaScript/issues/892)
**Pull Request:** [#904](https://github.com/frostney/GocciaScript/pull/904)

[ADR 0083](0083-realm-owned-prototype-method-hosts.md) migrated 7 prototype-host value types off a process-wide-pinned threadvar host to per-realm slots and, because that makes the host realm-owned (unpinned on realm `Destroy`), stopped caching their member definitions cross-realm — rebuilding them per realm. Its rationale notes that a cached cross-realm member array "would reference a freed instance" once the owning realm is torn down.

About 30 *other* value units (`Map`, `Set`, the weak collections, `Promise`, `ArrayBuffer`/`SharedArrayBuffer`, all `Intl*`, all `Temporal*`, `Headers`, `Response`, `TextDecoder`, the FFI values, `FinalizationRegistry`, ...) build their prototype via `TGocciaSharedPrototype.Create(Self)` — where `Self` is a per-realm host instance the realm unpins on `Destroy` — while still caching `FPrototypeMembers` cross-realm under `if Length(FPrototypeMembers) = 0`. They are therefore in the combination ADR 0083 calls unsafe: a realm-owned (freeable) host plus a cross-realm member cache whose `procedure of object` callbacks are bound to one realm's host. This ADR records the investigation of whether that is an exploitable use-after-free.

**It is not.** The engine invokes each cached callback as `callback(AArgs, AThisValue)`, and every bound callback re-derives its receiver from the JS `this` — `X := As<Type>(AThisValue, ...)` / `TGocciaXValue(AThisValue)` — and reads state only through that local; none is `virtual`. So when a later realm reuses the cache bound to an earlier, now-freed host, the freed host is passed as the implicit `Self` but is never dereferenced: not at the call (non-virtual → static code address, no VMT load) and not in the body (which uses `AThisValue`). A sweep of all 30 units confirmed this. The only two deviations — `TGocciaTemporalZonedDateTimeValue.ZonedDateTimeSubtract` and `TGocciaTemporalPlainYearMonthValue.YearMonthSubtract`, which call their sibling `…Add` unqualified (i.e. `Self.…Add(NewArgs, AThisValue)`) — still never dereference the host, because those siblings are likewise non-virtual and themselves re-bind `AThisValue`.

So ADR 0083's "would reference a freed instance" is literally true (the cached method pointer's data field *is* the freed host) but benign here: the reference is never read. ADR 0083's 7 units were migrated chiefly to fix a real **leak** — their hosts were pinned process-wide and never released — and the per-realm rebuild both dropped that permanent pin and made their safety independent of any "callbacks never touch the host" assumption. The 30 `TGocciaSharedPrototype` units have no such leak (their host is realm-owned and released on `Destroy`), so they were not migrated. Their continued safety rests on one invariant: **a bound prototype callback must derive its receiver from `AThisValue` and never read the bound host (`Self`).** That invariant is documented at the binding site (`Goccia.SharedPrototype`) and is locked in by `Goccia.SharedPrototypeRealmReuse.Test`, which runs many realms on one thread with a forced GC collection and a heap stomp between them, then exercises the shared-prototype types in the later realms; a callback that began dereferencing the host would crash or corrupt there.

A future option, should that invariant become hard to maintain, is to extend ADR 0083's per-realm member rebuild to these units as well (dropping the cross-realm cache). That is deliberately not done here: it is a large change across 30 hot units for no current correctness benefit, and the regression test plus the documented invariant guard the property directly.
